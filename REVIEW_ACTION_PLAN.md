# レビュー対応計画

作成日: 2026-07-19

原典: `REVIEW_FINDINGS.md`

対象基準: `main` (`29d497d`)

初期検証: Core 単体テスト 115/115 件成功、アプリは別出力先へのビルドで警告 0・エラー 0

## 1. 目的

第三者レビューで挙げられた指摘について、対応要否、優先度、採用する方針、完了条件を固定し、別の Codex タスクでも同じ判断基準で実装を継続できるようにする。

`REVIEW_FINDINGS.md` はレビュー原文として扱い、この文書を実装作業の進捗管理と意思決定の正とする。実装中に前提の誤りやより安全な方式が判明した場合は、コード変更より先にこの文書の該当項目へ理由を追記する。

## 2. 優先度

| 優先度 | 意味 |
|---|---|
| P0 | データ破壊や重大なセキュリティ問題。現時点では該当なし |
| P1 | 正式リリース前に対応する。取込漏れ、実環境での恒常的失敗、コピー復旧性に関係する |
| P2 | P1 後に対応する。誤操作、仕様不一致、診断性、運用上の大きな不便に関係する |
| P3 | 通常バックログ。軽微な不具合、性能、文書整合性、保守性の改善 |
| P4 | 条件付き対応または整理候補。現在の要件では実害がなく、必要になった時点で再評価する |

## 3. 共通の完了条件

各タスクは次をすべて満たした時点で完了とする。

- 対象外の指摘や無関係なリファクタリングを混在させない
- 仕様判断を `DESIGN.md`、`TEMPLATE_SPEC.md`、`README.md` の適切な箇所へ反映する
- 正常系だけでなく、レビューで指摘された失敗系の自動テストを追加する
- `dotnet test PhotoImporter.sln --no-restore` が成功する
- `PhotoImporter.App` が警告 0・エラー 0 でビルドできる
- 起動中の `PhotoImporter.exe` が通常出力先をロックしている場合は、プロセスを停止せず別の `OutDir` でビルドする
- 検証用に作成した出力や一時ファイルを残さない
- 本文の状態、実施内容、検証結果、残存リスクを更新する
- 1タスク1コミットを基本とし、コミットメッセージから対象IDを判別できるようにする

## 4. 実施順序

P1-A と P1-B はコピー処理周辺で変更箇所が重なる可能性があるため、並列実行しない。以下の順序を基本とする。

1. P1-A: コピー先ファイルシステムと取込済判定
2. P1-B: 読み取り専用ファイルとコピー防衛線テスト
3. P2-A: パス長とテンプレート事前検証
4. P2-B: 選択状態と再スキャン
5. P2-C: 対象ファイル列挙とドライブ利用不能時の制御
6. P2-D: 最終例外処理と残存一時ファイル案内
7. P3: 軽微な改善と文書整合
8. P4: 条件付き項目の再評価

## 5. 実装タスク

### [x] P1-A: コピー先ファイルシステムと取込済判定

対象: M-2、M-4

#### 採用判断

対応必須。M-4 は異なる写真を無言で「取込済」と判定する可能性があり、明示的なコピー失敗より優先度が高い。M-2 は FAT/FAT32/exFAT のコピー先で更新日時の精度差によりコピー検証や再スキャンが失敗する問題である。

#### 採用方針

- 更新日時比較を一律の Ticks 完全一致や一律 2 秒許容にしない
- コピー先ボリュームのファイルシステムに応じた比較ポリシーを導入する
  - NTFS/ReFS: 原則として完全一致
  - exFAT: 10ms 精度を考慮
  - FAT/FAT32: 2秒精度とローカル時刻保持の特性を考慮
  - 未知のファイルシステム: 安全側に不一致または明示的な非対応として扱う
- 一時ファイルと正式ファイルのコピー後検証にも同じ比較ポリシーを使用する
- `Imported` は少なくとも「サイズ一致」かつ「ファイルシステム精度を考慮した更新日時一致」の場合だけとする
- 現在の `source <= destination` だけによる取込済判定は廃止し、曖昧な場合は `Conflict` または連番候補へ回す
- 同一サイズ・同一時刻の別内容ファイルはサイズと時刻だけでは完全に識別できないことを残存リスクとして明記する
- ハッシュまたはカード識別子を含む取込履歴は、必要性と性能を別途評価し、このタスクへ無断で追加しない

#### 主な対象

- `src/PhotoImporter.Core/Copying/CopyEngine.cs`
- `src/PhotoImporter.Core/Templates/DestinationAllocator.cs`
- `src/PhotoImporter.Core/Metadata/VolumeInfo.cs`
- `src/PhotoImporter.App/MainWindow.xaml.cs`
- 関連するコピー計画・スナップショット型

#### 必須テスト

- 宛先サイズが異なる場合は、宛先時刻が新しくても `Imported` にならない
- 宛先サイズと時刻が一致する場合だけ `Imported` になる
- 宛先時刻が新しいだけでは `Imported` にならない
- NTFS 相当では許容範囲外の時刻差を拒否する
- exFAT 相当の 10ms 精度を考慮してコピー後検証できる
- FAT/FAT32 相当の 2秒精度を考慮してコピー後検証できる
- 未知のファイルシステムの扱いが仕様どおりである
- 再スキャン時に正常コピー済みファイルが再び上書き対象や競合にならない

#### 完了記録

- 実施内容: コピー先ボリュームから NTFS/ReFS・exFAT・FAT/FAT32 の更新日時比較ポリシーを決定し、取込済判定、コピー先再検証、一時ファイル検証、正式ファイル検証で共用するようにした。`Imported` はサイズとポリシー適用後の更新日時が両方一致する場合だけとし、それ以外は競合または連番候補へ回す。未知のファイルシステムはスキャン時に明示的な非対応エラーとする。`DESIGN.md`、`TEMPLATE_SPEC.md`、`README.md` も同じ仕様へ更新した。
- 検証結果: `dotnet test PhotoImporter.sln --no-restore` は 127 件成功。起動中アプリが通常出力先をロックしていたため、別 `OutDir` で `PhotoImporter.App` をビルドし、警告 0・エラー 0 を確認した。`git diff --check` も成功した。検証用 `OutDir` は削除済み。
- コミット: 未実施（コミット作成の依頼なし）。
- 残存リスク: 同一サイズ・同一更新日時の別内容ファイルは識別できない。ハッシュまたはカード識別子を含む取込履歴は別途評価が必要。FAT/FAT32・exFAT の実媒体でのコピーは未検証。

### [x] P1-B: 読み取り専用ファイルとコピー防衛線テスト

対象: M-1、T-1

#### 採用判断

対応必須。DCF プロテクト写真は実データで発生し、失敗時の `.partial` 残留と読み取り専用宛先の上書き不能を引き起こす。

#### 採用方針

- 既存の多層安全条件をすべて満たした、自アプリが作成した `.partial` だけを属性変更の対象とする
- 安全な `.partial` の削除直前に ReadOnly 属性だけを解除し、削除する
- 正式な宛先の読み取り専用属性は、最終再検証後かつ確定直前に限って解除する
- `MoveFileExW` が失敗し、旧宛先が残っている場合は読み取り専用属性の復元を試みる
- 属性復元を確認できない場合は通常エラーに丸めず、復旧が必要な状態として利用者へパスを示す
- コピー成功後はコピー元から引き継がれた属性を保持する
- 既存の一時ファイル削除条件、宛先再検証、TOCTOU 防御を緩めない

#### 主な対象

- `src/PhotoImporter.Core/Copying/CopyEngine.cs`
- `src/PhotoImporter.Core/Copying/CopyFile2Native.cs`
- `tests/PhotoImporter.Core.Tests/CopyEngineTests.cs`

#### 必須テスト

- 読み取り専用ソースのコピーに成功し、正式ファイルが期待する属性を持つ
- キャンセル時に読み取り専用 `.partial` が削除される
- リネーム失敗時に、安全条件を満たす読み取り専用 `.partial` が削除される
- 安全条件を満たさないファイルの属性を変更・削除しない
- 読み取り専用宛先の上書きに成功する
- 上書き失敗時に旧宛先の属性が復元される
- 属性復元を確認できない場合に復旧パスを含むエラーになる
- スキャン後の宛先出現を検出して非上書きコピーを中止する
- スキャン後の宛先変更を検出して上書きコピーを中止する
- `MoveFileExW` 失敗時の削除と保全が既存方針どおりに分岐する

#### 完了記録

- 実施内容: 安全条件を満たす現在処理中の `.partial` に限り、削除直前に ReadOnly 属性だけを解除するよう変更した。上書き先は最終再検証後かつ `MoveFileExW` 直前に ReadOnly を解除し、確定失敗時は旧宛先のスナップショット一致を確認して ReadOnly を復元・再確認する。復元または一時ファイル削除を確認できない場合は `CopyRecoveryException` として一時ファイルを保全し、旧宛先と一時ファイルの両パスを利用者へ示す。コピー・確定・属性操作にテスト用の内部境界を設け、読み取り専用コピー、キャンセル、確定失敗、属性復元失敗、宛先の途中出現・変更、安全条件不一致、一時ファイル削除・保全分岐を追加した。`DESIGN.md` と `README.md` も同じ仕様へ更新した。
- 検証結果: `dotnet test PhotoImporter.sln --no-restore -c Release` は 137 件成功。`dotnet build PhotoImporter.sln --no-restore -c Release` は警告 0・エラー 0。`git diff --check` も成功した。読み取り専用ソースと読み取り専用上書き先は実ファイル属性で検証し、`MoveFileExW` 失敗と属性復元失敗は決定的なテスト代替で検証した。
- コミット: 未実施（この作業ツリーに実装済み）
- 残存リスク: 最終再検証から ReadOnly 解除・`MoveFileExW` までの上書き TOCTOU 窓は Windows API 上原子的に閉じられず、D-7 の既知リスクとして残る。属性復元不能はテスト代替で検証しており、実デバイス固有の ACL・ウイルス対策ソフト・故障媒体による失敗は未検証。

### [x] P2-A: パス長とテンプレート事前検証

対象: M-3、M-5

#### 採用判断

対応する。どちらも安全側に失敗するが、本来は入力時に1回だけ示すべき問題がファイル単位のスキャンエラーとして表面化している。

#### 採用方針

- `DestinationAllocator` から `TemplateEvaluator.EvaluateDetailed` へ `destinationRoot` と決定した上限を渡す
- 長いパスをサポートする範囲と上限値を仕様で決定し、コードの既定値 259 と矛盾させない
- .NET Framework 4.8.1 に必要な `app.config` のパス処理設定を追加する
- Per-Monitor V2 DPI と `longPathAware` を含む manifest を追加し、プロジェクトへ接続する
- 日付書式をサンプル日時でパース時に検証する
- 日付書式の評価結果にパス不正文字が出るテンプレートを入力エラーとして扱う
- 先頭・末尾区切り、空パス要素など静的に判定できる構造エラーをパース時に返す
- ファイル固有値によってのみ発生するパスエラーは、引き続き行単位エラーとして扱う

#### 必須テスト

- 宛先ルート込みの上限直前・上限超過
- 長いルートと短い相対パスの組み合わせ
- `{ModifiedDate:HH:mm}` を入力時に拒否する
- 不正な日時書式を `InvalidDateFormat` として返す
- 先頭・末尾区切りと空要素を入力時に拒否する
- 正常な日時書式と複数階層テンプレートを誤拒否しない
- manifest/config を含むアプリビルドが成功する

#### 完了記録

- 実施内容: 宛先ルートと32,767文字上限を `DestinationAllocator` から全テンプレート評価へ渡し、ルート込みのフルパス長を検証するようにした。日時書式をサンプル日時でパース時に評価し、不正書式およびパス不正文字を生成する書式を `InvalidDateFormat` とした。先頭・末尾・連続区切り、静的な空要素、`.`、`..`、空の `{Sequence}` 要素を `InvalidPathStructure` として入力時に拒否する。長いパスと Per-Monitor V2 を有効にする App.config / manifest を追加してアプリへ接続し、DESIGN / TEMPLATE_SPEC に上限と前提を明記した。
- 検証結果: `dotnet test PhotoImporter.sln --no-restore -c Release` は146件成功。`dotnet build PhotoImporter.sln --no-restore -c Release` は警告0・エラー0。生成された `PhotoImporter.exe.config` に長いパス用 AppContext スイッチが含まれること、および `git diff --check` の成功を確認した。
- コミット: 未実施（この作業ツリーに実装済み）
- 残存リスク: 260文字超の実利用には Windows 側の LongPathsEnabled が必要であり、無効な端末ではOS/API側のエラーになる。ファイル固有の値でのみ発生する予約名、不正要素、各ファイルシステム固有の要素長制限は従来どおり行単位エラーとして扱う。

### [ ] P2-B: 選択状態と再スキャン

対象: M-7、T-2、D-9

#### 採用判断

対応する。コピー対象抽出は安全だが、チェックボックス表示とモデルが食い違い、再スキャン後に利用者が外した対象が再選択される。

#### 採用方針

- `DataGridCheckBoxColumn` の表示要素を `CanCopy` に連動して無効化する
- `IsSelected` が要求値を拒否した場合も、表示をモデルへ戻す通知を行う
- コピー後の自動再スキャンでは、コピー元相対パスをキーとして未処理行の選択状態を引き継ぐ
- コピーに成功して `Imported` になった行、競合行、エラー行は常に選択解除する
- 手動の新規スキャンで状態を引き継ぐかは仕様で明示し、コピー後再スキャンと混同しない

#### 必須テスト

- `CanCopy == false` の行へ `IsSelected = true` を設定しても false のままで通知される
- `SetCopyError` 後に選択解除され、再選択できない
- コピー可能行だけが初期選択される
- コピー後再スキャンで利用者が外した未処理行は外れたままになる
- コピー成功・競合・エラー行は再スキャン後も選択されない
- 無効行のチェックボックスが操作不能として表示される

### [ ] P2-C: 対象ファイル列挙とドライブ利用不能時の制御

対象: D-2、D-5

#### 採用判断

対応する。写真取込アプリとして対象外ファイルや既知のシステム領域を取り込まず、ドライブ全体が利用不能になった場合は同じエラーをファイル・フォルダごとに繰り返さない。

#### 採用方針

- 対応拡張子を単一のレジストリまたは判定関数に集約する
- 列挙、RAW+JPEG 判定、Exif 対応可否の拡張子定義を可能な限り共有する
- `Thumbs.db`、`desktop.ini` と既知のシステム領域を仕様に基づいて除外する
- 個別フォルダのアクセス拒否は既存どおり継続可能とする
- コピー元ルートまたはコピー先ルート自体が利用不能になった場合はバッチを中止し、単一の明確なエラーとして表示する
- ドライブ脱着を誤判定しないよう、個別ファイルエラーとの判定条件をテスト可能な形に分離する

#### 必須テスト

- 対応拡張子だけがプレビュー対象になる
- 大文字小文字の異なる拡張子を同じように扱う
- `Thumbs.db`、`desktop.ini`、既知のシステム領域を除外する
- 個別サブフォルダのアクセス拒否ではスキャンを継続する
- コピー元ルート利用不能ではスキャン全体を中止する
- コピー先ルート利用不能では残り全件を個別失敗にせずコピー全体を中止する

### [ ] P2-D: 最終例外処理と残存一時ファイル案内

対象: M-6、D-11

#### 採用判断

対応する。ただし主要な `Scan_Click` と `Copy_Click` には既に例外処理があるため、グローバルハンドラは通常制御ではなく最後の防衛線とする。

#### 採用方針

- WPF の未処理例外を記録し、利用者へエラーダイアログを表示する
- 未知の例外を握りつぶして処理継続せず、安全な終了を基本とする
- `TaskScheduler.UnobservedTaskException` は診断記録を目的とし、通常のエラー伝播の代替にしない
- 設定保存は閉じる時だけに依存せず、重要な設定確定時の保存を P3 の L-6 と連携して検討する
- 起動時にコピー先候補配下の厳密な命名規則に一致する `.partial` を検出する
- 検出時は自動的に成功・失敗を断定せず、正式ファイルとの状態、保存場所、利用者が取るべき操作を案内する
- 自動削除を追加する場合は既存の多層安全条件と同等以上の条件を要求する

#### 必須テスト

- UI スレッドの未処理例外が記録・表示され、安全な終了へ進む
- 通常のスキャン・コピー例外がグローバルハンドラへ漏れない
- 命名規則に一致する `.partial` だけを検出する
- リパースポイント、ディレクトリ、ルート外、名前不一致を復旧候補にしない
- 正式ファイルの有無とスナップショット状態に応じて案内を変える
