# 写真取込アプリ 検討メモ

作成日: 2026-07-12
最終更新日: 2026-07-14
状態: 開発ステップ4まで実装済み。ファイルシステム系テンプレートを使ったスキャン、判定、プレビュー、コピー実行が利用可能。技術スタックは **.NET Framework 4.8.1 + WPF**。

## 1. 目的・要件

Windows 用デスクトップアプリ。カメラ・SDカード・CFexpress カードから写真をPCへコピーして取り込む。

- コピー先のフォルダ・ファイル名を**テンプレート**で柔軟に設定できる
  - 例: `{TakenDate:yyyy-MM-dd}\{OriginalName}`
- コピー元とコピー先を比較し、**未取込の写真のみ**を判断してコピーする
- コピー元フォルダの直下だけでなく、**すべてのサブフォルダを再帰的に走査**する。デジタルカメラが `DCIM` 配下に複数の保存フォルダを作成する場合や、保存フォルダが複数階層になる場合も対象とする
- 使い方の流れ:
  1. 起動後、コピー元フォルダ・コピー先フォルダを選択し、テンプレートを入力
  2. 未取込の写真を一覧表示し、利用者が確認
  3. 未取込の写真をコピー
- 画面では左にコピー元ファイル、右にコピー先(予定)ファイルを並べ、どうコピーされるか視覚的に分かるようにする
- 未取込のみ/コピー済も含む全件、の表示切替ができる
- UI は**日本語・英語の2言語対応**とする。テンプレートのトークン名は表示言語によらず英語で統一する

### 配布要件

- **配布は exe のみ(単一ファイル)**
- ターゲットは Windows 11 に限定。.NET Framework 4.8.1 が OS 標準搭載のため、ランタイム同梱不要で exe を小さくできる
- WinUI 3 + Self-contained + Single-file は exe が数百MBになるため不採用

### net481 の既知の制約と対処

- **C# 言語バージョン**: 公式は C# 7.3。`LangVersion` を上げれば record / init も `IsExternalInit` ポリフィルで使用可。`Parallel.ForEachAsync` は無いので `SemaphoreSlim` + `Task.WhenAll` で並列化。`System.Text.Json` は NuGet 版が net481 で動く
- **High DPI**: Per-Monitor V2 は app.manifest / app.config での明示設定が必要
- **長いパス(260文字超)**: app.config の `Switch.System.IO.UseLegacyPathHandling=false` + OS 側 LongPathsEnabled。加えてアプリ側で宛先パス長のバリデーションを入れる
- **MTP 非対応(v1 の割り切り)**: カメラ本体の USB 接続(MTPデバイス)は通常のファイルAPIで読めない。v1 は**ドライブレターが付くもの(カードリーダー経由)に限定**。MTP は将来課題

### コピー元の再帰走査

- 利用者が選択したコピー元フォルダを**コピー元ルート**と呼ぶ。スキャン対象は、コピー元ルート直下およびその全子孫フォルダにある対応ファイルとする
- 各ファイルについて、コピー元ルートからの相対ファイルパスと相対ディレクトリパスを保持する。たとえばコピー元ルートがSDカードのルート `E:\`、実ファイルが `E:\DCIM\100MSDCF\DSC00001.ARW` の場合、相対ファイルパスは `DCIM\100MSDCF\DSC00001.ARW`、相対ディレクトリパスは `DCIM\100MSDCF` とする
- コピー元ルート直下のファイルでは、相対ディレクトリパスを空文字とする
- 列挙後はコピー元ルートからの相対ファイルパスを、大文字・小文字を区別しない序数比較で昇順に並べる。ファイルシステムから返される列挙順には依存しない
- ディレクトリのシンボリックリンク、ジャンクションなどのリパースポイントは追跡しない。循環参照とコピー元ルート外の走査を防ぐ
- 読み取れないフォルダや、走査中に削除・抜去されたファイルはスキャンエラーとして記録し、利用者へ対象パスを表示する。カード全体を読み取れなくなった場合はスキャンを中止する
- コピー元とコピー先が同一、または一方が他方の配下になる組み合わせは禁止する。コピー結果を再帰走査で再びコピー元として検出することを防ぐ
- 対応拡張子によるファイル種別フィルタは、再帰列挙したすべてのファイルへ同じ規則で適用する

## 3. テンプレート仕様(v1方針確定)

詳細は [TEMPLATE_SPEC.md](TEMPLATE_SPEC.md) を正とする。

- トークン名は表示言語によらず英語の PascalCase で記述する
- パス区切り `\` を含めて、コピー先ルートからの相対パスを記述する
- 基本形は `{Token}`、書式対応トークンは `{Token:Format}`
- 既定テンプレートは `{TakenDate:yyyy-MM-dd}\{OriginalName}`
- v1 のファイルシステム系トークンは `{OriginalName}` `{FileName}` `{Extension}` `{SourceRelativeDirectory}` `{ModifiedDate}` `{FileSize}` `{Sequence}`
- v1 の Exif 系トークンは `{TakenDate}` `{TakenDateLocal}` `{TakenDateInTimeZone}` `{CameraMake}` `{CameraModel}` `{Lens}`
- Exif撮影日時は、タイムゾーン未考慮、PCのタイムゾーン、指定したタイムゾーンの3方式を選べる。指定には `JST` などアプリ定義の3文字コード、または `UTC+9` などUTCからの固定時差を使用する
- `{Sequence}` は競合がなければ空文字、競合時は既定3桁の `_001` から付与する。`{Sequence:4}` のように桁数を指定できる

トークンは2分類:

| 分類 | トークン例 | コスト |
|---|---|---|
| ファイルシステム系 | `{OriginalName}` `{FileName}` `{Extension}` `{SourceRelativeDirectory}` `{ModifiedDate:書式}` `{FileSize}` `{Sequence}` | `{Sequence}` の競合確認以外はゼロ(列挙時に取得済み) |
| Exif系 | `{TakenDate:書式}` `{TakenDateLocal:書式}` `{TakenDateInTimeZone:JST}` `{TakenDateInTimeZone:UTC+9}` `{CameraMake}` `{CameraModel}` `{Lens}` | ヘッダ読み込みが必要 |

- テンプレート解析時点で「Exif系トークンを含むか」が静的に判定できる → ファイルシステム系のみなら Exif を読まずに全宛先パスを即計算できる
- Exif が無いファイル(動画・一部RAW)では各撮影日時トークンを更新日時へフォールバックする
- 日時書式は .NET のカスタム日時書式を `InvariantCulture` で解釈する
- テンプレートから生成されるパスは必ず相対パスとし、絶対パス・親ディレクトリ参照・不正文字を拒否する
- `{SourceRelativeDirectory}` はコピー元ルートから元ファイル格納フォルダまでの相対パスを展開し、コピー元の複数階層をコピー先へ引き継げる。詳細な空文字時の区切り規則は `TEMPLATE_SPEC.md` に従う
- UI上のトークン説明、入力エラー、プレビュー状態は日本語・英語にローカライズする

## 4. 未取込判定の設計

基本方針: **テンプレートから宛先パスを計算し、宛先にファイルが存在して、コピー元の更新日時がコピー先と同じか古ければ取込済**。日時比較には `LastWriteTimeUtc` を使用する。

- 宛先フォルダの実ファイルを真実(source of truth)とする。ユーザーが宛先を手で整理しても、別PCでコピーしても破綻しない
- 「取込済台帳(コピー記録DB)」方式は、テンプレート変更や宛先の手動整理で実態とズレるため不採用

### Exif 系テンプレート使用時の高速化: Exif キャッシュ

基本フロー:

1. カード挿入 → ファイル列挙(速い)→ キャッシュヒット分は Exif 読み込みスキップ
2. キャッシュミス分のみ並列(4〜8並列)でヘッダ読み込み

同じカードの使い回しでは2回目以降、新規撮影分しか Exif を読まない。初回も 1000枚 × ヘッダ数十KB なら数秒〜十数秒のオーダー(進捗バー表示)。

#### キーと同一性判定

- エントリのキーは `(ボリュームシリアル番号, ボリュームルートからの相対パス, ファイルサイズ, LastWriteTimeUtc)` の**4要素すべて**とする。キーが完全一致した場合のみキャッシュヒットとし、不一致なら上書きせず別エントリとして追加する(追記型)。複数カードに同一パス・同一ファイル名のファイルが存在しても、互いのキャッシュを壊さない
- 相対パスの基準はコピー元ルートではなく**ボリュームルート**とする。コピー元ルートの選び方(`E:\` と `E:\DCIM` など)が変わってもキーが変わらないようにする。大文字・小文字は序数比較で正規化する
- 更新日時は UTC の Ticks で保持し、タイムゾーン・DST の影響を受けないようにする
- ボリュームシリアル番号は `GetVolumeInformationW` の P/Invoke で取得する。シリアルはフォーマット時に付与されるため、**カードを再フォーマットすると別カードとして認識される**。これは仕様として割り切り、UI で「カードを再フォーマットすると新しいカードとして認識されます」と説明する
- セクタ単位クローンによるシリアル重複は、キーにサイズと更新日時が含まれるため誤った Exif を返すことはない。カードの名前・縮小版が混ざる可能性は割り切る
- 値: 撮影日時・撮影日時オフセット・カメラメーカー・カメラモデル・レンズなどの抽出済み Exif と、エントリの最終利用日
- キャッシュヒット時はエントリの最終利用日を更新し、スキャン完了時にまとめて書き出す

#### 保存形式

- 保存場所は `%LocalAppData%\PhotoImporter\ExifCache\` とし、**ボリュームシリアル番号ごとに1フォルダ**を作る

```
%LocalAppData%\PhotoImporter\ExifCache\
  A1B2C3D4\
    meta.json      … 利用者が付けた名前、ボリュームラベル、容量、ファイルシステム、
                     ドライブ種別、初回利用日、最終利用日、エントリ件数、スキーマバージョン
    entries.json   … Exif エントリ本体(数千件になりうる)、スキーマバージョン
    thumb_*.jpg    … 縮小版(埋め込みサムネイルの抜き出し)
```

- カード単位削除=フォルダ削除、キャッシュ容量表示=フォルダ内ファイルサイズ合計、と管理操作をファイル操作に1対1で対応させる
- 起動時は各カードの meta.json のみ読む。entries.json はスキャン対象ボリュームの分だけ遅延読み込みし、カードが増えても起動が遅くならないようにする
- 書き込みは一時ファイル→リネームで確定する(コピー処理と同じ流儀)。JSON が壊れていた場合は破棄して再生成する(キャッシュであり、失っても取込判定の正しさに影響しない)
- meta.json と entries.json のスキーマバージョンは独立に持ち、抽出項目の追加時に旧エントリを無効化できるようにする

#### カード管理画面

- キャッシュのあるカードを一覧表示する。表示項目: 名前、ボリュームラベル、容量、ドライブ種別、最終利用日、キャッシュ容量、エントリ件数、縮小版
- 利用者は目印となる名前を付けられる
- 現在コピー元に選択中のボリュームとシリアルが一致するカードを一覧上でハイライトする。ハイライトはキャッシュ設定 OFF でも動作する(シリアル照合のみで、キャッシュの読み書きを伴わないため)
- ドライブ種別(リムーバブル・固定・ネットワーク)はスキャン時に `GetDriveType` で記録し、カード以外のボリューム(HDD・NAS 等をコピー元にした場合)も区別して表示する。一覧からは除外せず、見えて消せる状態にする
- 削除は**手動のみ**とし、自動削除はしない:
  - カード単位の削除(フォルダごと)
  - カード内の「最後に使われてから n 日より古いエントリ」の削除。基準は撮影日時ではなく**エントリの最終利用日**とする。撮影日時基準では、カード上に残っている古い写真のエントリが削除→次回スキャンで再読み込み→再キャッシュのループになり、容量削減にならないため。UI 文言も「最後に使われてから n 日」であることが伝わる表現にする
- 削除確認では「キャッシュを消しても写真は消えない。次回スキャンが遅くなるだけ」を明記し、利用者が安心して消せるようにする

#### 縮小版(サムネイル)

- Exif 埋め込みサムネイル(IFD1 の 160×120 程度の JPEG)を、ヘッダ読み込みのついでに抜き出してそのまま保存する。画像本体(RAW 含む)の自前デコードはしない
- **撮影日時の古い方 N 枚+新しい方 N 枚**を保持する。最初に撮った写真でどのカードかを覚えていることがあるため、古い方も持つ
- スキャン完了時に「今回のスキャンに存在するファイル」の中から候補を選び直す。候補にサムネイル未保存のファイル(過去のスキャンでキャッシュ済みのため今回ヘッダを読まなかったファイル)が入った場合は、そのファイルだけヘッダを追加で再読み込みする(最大 2N 件で実質コストゼロ)
- 埋め込みサムネイルを取れないファイル(動画など)はスキップし、取れるファイルから選ぶ
- 古い方と新しい方の候補が重複した場合は重複排除する(枚数が少ないカードで起こる)
- 撮影日時が無いファイルは更新日時にフォールバックする(Exif 系トークンと同じ方針)
- 縮小版は写真の内容が PC 側に残ることを意味する。カードから写真を消しても縮小版は残るため、削除 UI(カード単位削除・設定 OFF 時の削除)で確実に消えることと対にする

#### 設定

- **「Exif キャッシュを利用する」**: OFF の場合、読み取り・書き込み・メタ更新のすべてを行わない(読み取りだけ許すと、最終利用日の更新という書き込みが必要になり中途半端な状態になるため)。OFF でも管理画面と削除機能は利用できる(既存キャッシュを消すため)
- **「縮小版を保持する」**: キャッシュ設定に従属し、キャッシュ OFF 時は設定不可(グレーアウト)とする。ON→OFF に切り替えたときは、確認のうえ既存の縮小版も削除する(OFF にする動機はプライバシーである可能性が高く、過去のサムネイルが残り続けるのは期待に反するため)
- Exif 系トークンを含まないテンプレートでのスキャンは Exif を読まないため、キャッシュも縮小版も作成・更新しない。カードが管理画面に現れるのは Exif 系テンプレートで一度でもスキャンした後になる。**「最終利用日」は「キャッシュを最後に利用した日」と定義**し、この挙動と一貫させる

### 判定の3状態

- **未取込**: 宛先に無い。`{Sequence}` がある場合は、必要に応じて既定3桁の `_001` から空いている番号を付与した宛先を使用する
- **取込済**: 宛先に有り、コピー元の更新日時がコピー先と同じか古い。ファイルサイズは表示と変更検知に使用するが、この判定では更新日時を優先する
- **競合**: コピー元の方が新しく、`{Sequence}` がなく、かつ上書きしない設定の場合。または指定された桁数で表現できる連番を使い切った場合

### コピー実行方針

- スキャン時に、表示する宛先相対パス、コピー元のサイズと `LastWriteTimeUtc`、コピー先の存在・サイズ・`LastWriteTimeUtc` をコピー計画として保持する
- コピー時にテンプレートを再評価したり、別の宛先を再割り当てしたりしない。利用者が確認したプレビューと実際のコピー先を一致させる
- コピー直前にはコピー計画の再計算ではなく、コピー元とコピー先がスキャン時から変わっていないかを再検証する。変化していたファイルはコピーせず、再スキャンが必要なエラーとして表示する
- UI に「既存ファイルを上書きする」設定を設ける。設定はスキャン前に選択し、プレビューの割り当てと状態へ反映する
  - `{Sequence}` がある: 上書き設定を参照しない。コピー元の方が新しくコピーが必要なら、常に次の連番候補を表示済み宛先として割り当てる
  - `{Sequence}` がない・上書きする: コピー元の方が新しければ、表示済みの同じ宛先へコピーする
  - `{Sequence}` がない・上書きしない: コピー元の方が新しければ競合とする
- コピー元の更新日時がコピー先と同じか古い場合は、上書き設定にかかわらずコピー対象外とする
- コピー元は削除・変更せず、ファイルは1件ずつ処理する

#### Windows API と一時ファイル

- ファイル内容のコピーには、Windows 8以降が提供する `CopyFile2` を使用する。対象OSはWindows 11のためフォールバックは設けず、アプリ独自のバッファ読み書きは実装しない
- コピー先と同じフォルダに `PI_<32桁のGUID>.partial` 形式の一時ファイルを作る。先頭にドットは付けず、`PI_` をアプリ識別子とする
- `COPYFILE2_EXTENDED_PARAMETERS` には `COPY_FILE_FAIL_IF_EXISTS`、キャンセル情報、進捗コールバック、コールバックコンテキストを指定する。一時名はGUIDで一意にするが、万一の衝突でも既存ファイルを上書きしない
- `COPYFILE2_CALLBACK_STREAM_FINISHED` を受けた時点で、メッセージの `Info.StreamFinished.hDestinationFile` に対して直ちに `FlushFileBuffers` を呼ぶ。ハンドルが有効なコールバック内で実行し、アプリ側では `CloseHandle`、保持・複製・再オープンをしない。これにより `CreateFileW`、読み取り専用属性の一時解除、再オープンに伴う時刻への懸念をなくす
- `FlushFileBuffers` が失敗した場合は、コールバックコンテキストへWin32エラーを保存して `COPYFILE2_PROGRESS_CANCEL` を返す。マネージ例外をネイティブコールバック境界の外へ送出しない。`CopyFile2` が返った後、保存したフラッシュエラーをファイル単位の失敗として報告し、正式名へ移さない
- `COPYFILE2_CALLBACK_CHUNK_FINISHED` の `uliTotalBytesTransferred` を全体バイト進捗へ使用する。利用者のキャンセル要求を検出した場合も `COPYFILE2_PROGRESS_CANCEL` を返し、Windowsにコピー断片の削除を要求する。`COPYFILE2_PROGRESS_STOP` と `COPYFILE2_PROGRESS_PAUSE` は使用しない
- `CopyFile2` の `S_OK` はコピー処理の完了を示すが、`FlushFileBuffers` を含めても記憶装置が要求を正しく実装しない場合や電源断に対する絶対保証とはみなさない。`COPY_FILE_NO_BUFFERING` は指定せず、各ストリーム完了時のフラッシュを使用する
- `CopyFile2` は拡張属性、OLE構造化ストレージ、NTFS代替データストリーム、ファイル属性を保持する一方、既定ではコピー元のセキュリティ属性をコピーしない。本アプリではコピー先フォルダ側のアクセス制御を適用したいので、この動作を採用し `COPY_FILE_REQUEST_SECURITY_PRIVILEGES` は指定しない
- `CopyFile2` が成功した後に一時ファイルのサイズと `LastWriteTimeUtc` をスキャン時のコピー元情報と照合してから正式名の確定へ進む。最終アクセス日時は取込判定やコピー検証には使用しない
- 一時ファイルへのコピー完了後、Windows の `MoveFileExW` で表示済みの正式名へ確定する。同じボリューム内で確定できるよう、一時ファイルは必ず正式なコピー先と同じフォルダに置く
- 上書きする場合は `MOVEFILE_REPLACE_EXISTING | MOVEFILE_WRITE_THROUGH`、上書きしない場合は `MOVEFILE_WRITE_THROUGH` を使用する
- `ReplaceFileW` は使用しない。このAPIは置換前ファイルの作成日時、短いファイル名、オブジェクトID、DACL、暗号化・圧縮属性、名前付きストリームなどを置換ファイルへ引き継ぐ「既存ファイルの更新」に適している。本アプリでは、コピー元から作成した一時ファイルの内容とメタデータを正式な宛先として確定し、古いコピー先固有のメタデータを引き継がない方がコピー操作の意味に合う
- `ReplaceFileW` は置換先が存在する場合専用であり、新規確定には別APIが必要になる。また `REPLACEFILE_WRITE_THROUGH` はWindowsでサポートされておらず、一部の失敗コードでは置換元・置換先の名前や継承済みメタデータが部分的に変化し得る。新規確定と上書き確定の両方を `MoveFileExW` に統一して、分岐と復旧処理を単純にする
- `MOVEFILE_WRITE_THROUGH` を電源断に対する完全な確定保証とはみなさない。Windowsの公開仕様が明示しているフラッシュ保証は、移動がコピーと削除として実行される場合に対するものであり、同一ボリュームの置換を含む名前空間更新のクラッシュ原子性までは保証していない
- `MoveFileExW` が通常の実行中に失敗を返した場合は、正式ファイルと一時ファイルの両方を再確認する。正式ファイルがスキャン時の旧状態のままなら一時ファイルを安全条件に従って削除する。正式ファイルが無い、または状態を判定できない場合は一時ファイルを自動削除せず、復旧が必要なエラーとして残す
- 電源断・OSクラッシュではアプリが `MoveFileExW` の戻り値を受け取れないため、次回起動時に名前だけからコピーの成否を決めない。次の状態を確認して扱う
  - 正式ファイルだけがあり、サイズと更新日時がコピー元の期待値と一致する: コピー完了として扱う
  - スキャン時の旧正式ファイルと一時ファイルがある: 正式名の確定前に停止した可能性がある。コピー元を再スキャンし、必要なら元ファイルからコピーをやり直す
  - 正式ファイルと一時ファイルの両方がある: どちらが期待する内容かを名前だけでは判断せず、正式ファイルをコピー元と比較して再判定する
  - 正式ファイルがなく一時ファイルだけがある: 一時ファイルが唯一残ったコピー先データの可能性があるため、自動削除しない。また、一時名には本来の正式名やコピー元との対応情報がないため、自動的に正式名へ変更もしない。復旧が必要なエラーとして利用者へ表示する
  - 正式ファイルも一時ファイルも期待状態でない: コピー元が利用可能なら、コピー元を正として再スキャン・再コピーする
- 起動時に `PI_<32桁のGUID>.partial` を一括削除しない。「再スキャン・再コピー」とは、残った一時ファイルをコピー元として使うのではなく、削除していない元写真からコピー計画を作り直し、必要なファイルをもう一度コピーすることを指す。コピー元が利用できない場合は一時ファイルを保全し、自動処理せず利用者へ通知する
- `MoveFileExW` が成功を返した後も、正式ファイルが存在してスキャン時のコピー元サイズおよび更新日時と一致し、一時ファイルが存在しないことを確認してから、そのファイルを完了扱いにする
- キャンセルまたは失敗時に削除できるのは、現在のコピー処理が作成したパスとして記録され、コピー先ルート配下にあり、ファイル名が厳密に `^PI_[0-9A-Fa-f]{32}\.partial$` と一致する通常ファイルだけとする。`PI_` を含むだけのファイルや、利用者の通常ファイルは削除しない

#### 進捗とエラー

- 件数は「確定済みファイル数 / コピー対象ファイル数」を表示する
- バイト数は `CopyFile2` の `COPYFILE2_CALLBACK_CHUNK_FINISHED` を使い、「全完了ファイルの合計バイト数 + 現在コピー中の転送済みバイト数 / コピー対象の合計バイト数」を表示する
- 現在のファイル単独の進捗率は表示しない。UI には全体の件数と全体のバイト数だけを表示する
- キャンセル時は現在の Windows コピー処理を中止し、安全条件を満たす一時ファイルを削除する。確定済みファイルは残し、未処理ファイルには手を付けない
- ファイル単位のエラーは記録して、コピー元全体が利用不能になった場合を除き、残りの処理を続行する
- コピー完了後、一時ファイルと正式ファイルのサイズおよび `LastWriteTimeUtc` がコピー元のスキャン時情報と一致することを確認する。ハッシュ検証は仕上げ段階の追加候補とする

## 5. 画面イメージ

```
[コピー元フォルダ選択] [コピー先フォルダ選択]
[テンプレート入力欄] [スキャン]
[既存ファイルを上書きする] [コピー] [キャンセル]
─────────────────────────────
 対象  コピー元                →   コピー先(予定パス)      状態
  ☑   DCIM\100MSDCF\DSC00001.ARW → 2026-07-12\DSC00001.ARW 未取込
  ☐   DCIM\101MSDCF\DSC00002.ARW → 2026-07-11\DSC00002.ARW 取込済
─────────────────────────────
 [全体進捗バー]  完了件数 / 対象件数  転送済み / 合計バイト数
```

- 左右独立2ペインではなく、**1行に「元パス → 宛先パス → 状態」を並べる一覧**の方が対応関係が明確で実装も簡単
- コピー元欄にはファイル名だけでなく、コピー元ルートからの相対ファイルパスを表示する。異なるサブフォルダに同名ファイルが存在しても識別できるようにする
- テンプレートや上書き設定を変更した場合は、再スキャンしてコピー計画を更新する
- 数千行は WPF の UI 仮想化(VirtualizingStackPanel)で問題なし
- 状態フィルターは未実装。今後、未取込のみ・全件などの表示切替を追加する
