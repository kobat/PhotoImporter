# 写真取込アプリ 検討メモ

作成日: 2026-07-12
状態: 設計検討フェーズ(実装未着手)。技術スタックは **.NET Framework 4.8.1 + WPF** に決定済み。

## 1. 目的・要件

Windows 用デスクトップアプリ。カメラ・SDカード・CFexpress カードから写真をPCへコピーして取り込む。

- コピー先のフォルダ・ファイル名を**テンプレート**で柔軟に設定できる
  - 例: `{TakenDate:yyyy-MM-dd}\{OriginalName}`
- コピー元とコピー先を比較し、**未取込の写真のみ**を判断してコピーする
- コピー元フォルダの直下だけでなく、**すべてのサブフォルダを再帰的に走査**する。デジタルカメラが `DCIM` 配下に複数の保存フォルダを作成する場合や、保存フォルダが複数階層になる場合も対象とする
- 使い方の流れ:
  1. 起動後、コピー元フォルダ・コピー先フォルダを選択し、テンプレートを入力
  2. 未取込の写真を一覧表示し、利用者が確認
  3. 未取込の写真をコピー
- 画面では左にコピー元ファイル、右にコピー先(予定)ファイルを並べ、どうコピーされるか視覚的に分かるようにする
- 未取込のみ/コピー済も含む全件、の表示切替ができる
- UI は**日本語・英語の2言語対応**とする。テンプレートのトークン名は表示言語によらず英語で統一する

### 配布要件

- **配布は exe のみ(単一ファイル)**
- ターゲットは Windows 11 に限定。.NET Framework 4.8.1 が OS 標準搭載のため、ランタイム同梱不要で exe を小さくできる
- WinUI 3 + Self-contained + Single-file は exe が数百MBになるため不採用

## 2. 技術スタック(決定済み)

| 項目 | 選定 | 理由・備考 |
|---|---|---|
| フレームワーク | .NET Framework 4.8.1 + **WPF** | Windows 11 標準のランタイムを利用でき、exe が小さい。WinForms より XAML レイアウト・DPI対応・バインディングが快適 |
| csproj | SDK スタイル + `<TargetFramework>net481</TargetFramework>` | `dotnet build` 可、NuGet は PackageReference |
| Exif 読み取り | **MetadataExtractor**(NuGet) | ヘッダ部分のみ読むため高速。net462+ 対応で net481 で動作 |
| Exif キャッシュ | **JSON ファイル1つ**(起動時全読み・終了時書き出し) | SQLite はネイティブDLLを伴い単一exe化と相性が悪いので不採用。件数が増えて困れば LiteDB(純マネージド)に切替 |
| 単一 exe 化 | **Costura.Fody**(依存DLLをリソース埋め込み)または ILRepack | マネージドDLLのみなら安定 |
| フォルダ選択 | `IFileOpenDialog` の COM 相互運用ラッパー(50行程度) | net481 標準の FolderBrowserDialog は古いツリー型のため |
| 構成 | コアロジック(テンプレート解析・差分判定・キャッシュ)は **UI と分離したクラスライブラリ** + 単体テスト(xUnit等) | 一番バグりやすく一番テストしやすい部分 |
| UI 多言語化 | `.resx` リソース(`ja-JP` / `en-US`) | UI文言とエラーメッセージをローカライズ。テンプレート構文と保存値は言語非依存 |

### net481 の既知の制約と対処

- **C# 言語バージョン**: 公式は C# 7.3。`LangVersion` を上げれば record / init も `IsExternalInit` ポリフィルで使用可。`Parallel.ForEachAsync` は無いので `SemaphoreSlim` + `Task.WhenAll` で並列化。`System.Text.Json` は NuGet 版が net481 で動く
- **High DPI**: Per-Monitor V2 は app.manifest / app.config での明示設定が必要
- **長いパス(260文字超)**: app.config の `Switch.System.IO.UseLegacyPathHandling=false` + OS 側 LongPathsEnabled。加えてアプリ側で宛先パス長のバリデーションを入れる
- **MTP 非対応(v1 の割り切り)**: カメラ本体の USB 接続(MTPデバイス)は通常のファイルAPIで読めない。v1 は**ドライブレターが付くもの(カードリーダー経由)に限定**。MTP は将来課題

### コピー元の再帰走査

- 利用者が選択したコピー元フォルダを**コピー元ルート**と呼ぶ。スキャン対象は、コピー元ルート直下およびその全子孫フォルダにある対応ファイルとする
- 各ファイルについて、コピー元ルートからの相対ファイルパスと相対ディレクトリパスを保持する。たとえばコピー元ルートがSDカードのルート `E:\`、実ファイルが `E:\DCIM\100MSDCF\DSC00001.ARW` の場合、相対ファイルパスは `DCIM\100MSDCF\DSC00001.ARW`、相対ディレクトリパスは `DCIM\100MSDCF` とする
- コピー元ルート直下のファイルでは、相対ディレクトリパスを空文字とする
- 列挙後はコピー元ルートからの相対ファイルパスを、大文字・小文字を区別しない序数比較で昇順に並べる。ファイルシステムから返される列挙順には依存しない
- ディレクトリのシンボリックリンク、ジャンクションなどのリパースポイントは追跡しない。循環参照とコピー元ルート外の走査を防ぐ
- 読み取れないフォルダや、走査中に削除・抜去されたファイルはスキャンエラーとして記録し、利用者へ対象パスを表示する。カード全体を読み取れなくなった場合はスキャンを中止する
- コピー元とコピー先が同一、または一方が他方の配下になる組み合わせは禁止する。コピー結果を再帰走査で再びコピー元として検出することを防ぐ
- 対応拡張子によるファイル種別フィルタは、再帰列挙したすべてのファイルへ同じ規則で適用する

## 3. テンプレート仕様(v1方針確定)

詳細は [TEMPLATE_SPEC.md](TEMPLATE_SPEC.md) を正とする。

- トークン名は表示言語によらず英語の PascalCase で記述する
- パス区切り `\` を含めて、コピー先ルートからの相対パスを記述する
- 基本形は `{Token}`、書式対応トークンは `{Token:Format}`
- 既定テンプレートは `{TakenDate:yyyy-MM-dd}\{OriginalName}`
- v1 のファイルシステム系トークンは `{OriginalName}` `{FileName}` `{Extension}` `{SourceRelativeDirectory}` `{ModifiedDate}` `{FileSize}` `{Sequence}`
- v1 の Exif 系トークンは `{TakenDate}` `{TakenDateLocal}` `{TakenDateInTimeZone}` `{CameraMake}` `{CameraModel}` `{Lens}`
- Exif撮影日時は、タイムゾーン未考慮、PCのタイムゾーン、指定したタイムゾーンの3方式を選べる。指定には `JST` などアプリ定義の3文字コード、または `UTC+9` などUTCからの固定時差を使用する
- `{Sequence}` は競合がなければ空文字、競合時は既定3桁の `_001` から付与する。`{Sequence:4}` のように桁数を指定できる

トークンは2分類:

| 分類 | トークン例 | コスト |
|---|---|---|
| ファイルシステム系 | `{OriginalName}` `{FileName}` `{Extension}` `{SourceRelativeDirectory}` `{ModifiedDate:書式}` `{FileSize}` `{Sequence}` | `{Sequence}` の競合確認以外はゼロ(列挙時に取得済み) |
| Exif系 | `{TakenDate:書式}` `{TakenDateLocal:書式}` `{TakenDateInTimeZone:JST}` `{TakenDateInTimeZone:UTC+9}` `{CameraMake}` `{CameraModel}` `{Lens}` | ヘッダ読み込みが必要 |

- テンプレート解析時点で「Exif系トークンを含むか」が静的に判定できる → ファイルシステム系のみなら Exif を読まずに全宛先パスを即計算できる
- Exif が無いファイル(動画・一部RAW)では各撮影日時トークンを更新日時へフォールバックする
- 日時書式は .NET のカスタム日時書式を `InvariantCulture` で解釈する
- テンプレートから生成されるパスは必ず相対パスとし、絶対パス・親ディレクトリ参照・不正文字を拒否する
- `{SourceRelativeDirectory}` はコピー元ルートから元ファイル格納フォルダまでの相対パスを展開し、コピー元の複数階層をコピー先へ引き継げる。詳細な空文字時の区切り規則は `TEMPLATE_SPEC.md` に従う
- UI上のトークン説明、入力エラー、プレビュー状態は日本語・英語にローカライズする

## 4. 未取込判定の設計

基本方針: **テンプレートから宛先パスを計算し、宛先にファイルが存在し、かつサイズが一致すれば取込済**。

- 宛先フォルダの実ファイルを真実(source of truth)とする。ユーザーが宛先を手で整理しても、別PCでコピーしても破綻しない
- 「取込済台帳(コピー記録DB)」方式は、テンプレート変更や宛先の手動整理で実態とズレるため不採用

### Exif 系テンプレート使用時の高速化: Exif キャッシュ

1. キャッシュのキー: `(ボリュームシリアル番号, 相対パス, ファイルサイズ, 更新日時)` → 値: 撮影日時・撮影日時オフセット・カメラ名などの抽出済み Exif
2. カード挿入 → ファイル列挙(速い)→ キャッシュヒット分は Exif 読み込みスキップ
3. キャッシュミス分のみ並列(4〜8並列)でヘッダ読み込み

同じカードの使い回しでは2回目以降、新規撮影分しか Exif を読まない。初回も 1000枚 × ヘッダ数十KB なら数秒〜十数秒のオーダー(進捗バー表示)。

### 判定の3状態

- **未取込**: 宛先に無い。`{Sequence}` がある場合は、必要に応じて既定3桁の `_001` から空いている番号を付与した宛先を使用する
- **取込済**: 宛先に有り・サイズ一致
- **競合**: `{Sequence}` がない場合に、宛先に同名だがサイズ不一致。または指定された桁数で表現できる連番を使い切った場合

## 5. 画面イメージ

```
[コピー元フォルダ選択] [コピー先フォルダ選択]
[テンプレート入力欄] (入力中にリアルタイムでプレビュー反映)
[フィルタ: ●未取込のみ ○すべて]  [スキャン]
─────────────────────────────
 コピー元                →   コピー先(予定パス)      状態
 DCIM\100MSDCF\DSC00001.ARW → 2026-07-12\DSC00001.ARW 未取込
 DCIM\101MSDCF\DSC00002.ARW → 2026-07-11\DSC00002.ARW 取込済
─────────────────────────────
 [選択した N 件をコピー]  進捗バー
```

- 左右独立2ペインではなく、**1行に「元パス → 宛先パス → 状態」を並べる一覧**の方が対応関係が明確で実装も簡単
- コピー元欄にはファイル名だけでなく、コピー元ルートからの相対ファイルパスを表示する。異なるサブフォルダに同名ファイルが存在しても識別できるようにする
- テンプレート編集のたびに(キャッシュがあるので)プレビューを即再計算できる
- 数千行は WPF の UI 仮想化(VirtualizingStackPanel)で問題なし

## 6. 開発ステップ(1ステップずつ動作確認しながら進める)

1. **コアロジック先行(UIなし)**: テンプレートのパーサと宛先パス計算をクラスライブラリ+単体テストで作成。まずファイルシステム系トークンのみ。`{SourceRelativeDirectory}` と複数階層のパス評価を含め、トークン名等の仕様をここで確定
2. **最小UI**: WPF プロジェクト作成、フォルダ選択2つ+テンプレート入力+再帰走査した一覧表示(全件、状態判定なし)。相対ファイルパスの表示、リパースポイント非追跡、読み取りエラー、コピー元・コピー先の包含関係の拒否を確認
3. **未取込判定(ファイルシステム系のみ)**: 宛先存在チェック+3状態表示+フィルタ
4. **コピー実行**: 進捗表示、キャンセル、エラー処理(カード抜去など)。→ この時点で Exif 不使用テンプレートなら実用可能
5. **Exif 対応**: MetadataExtractor 導入、`{TakenDate}` 等のExif系トークンを評価可能にする。テスト用写真の実データで検証
6. **Exif キャッシュ**: JSON キャッシュで2回目以降を高速化。実測して効果確認
7. **仕上げ**: 設定保存(前回のフォルダ・テンプレート)、テンプレートプリセット、コピー後ベリファイ(サイズ/ハッシュ)、単一 exe 化(Costura)の組み込み
