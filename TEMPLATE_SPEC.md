# 写真取込テンプレート仕様

作成日: 2026-07-12  
最終更新日: 2026-07-16  
対象バージョン: v1  
状態: 実装開始可能。2026-07-16 に Exif 拡張トークン(寸法・向き・露出・焦点距離・評価・GPS・シリアル)、`{Protected}`、数値書式・シャッター書式・GPS書式を追加(実装はこれから)

## 1. 目的

テンプレートから、コピー先ルートフォルダを基準とする相対パスを生成する。

テンプレート構文と保存値はUIの表示言語から独立させる。UIが日本語でも英語でも、トークンは常に本書で定義する英語名を使用する。UI上の説明、検証メッセージ、状態表示のみをローカライズする。

既定テンプレート:

```text
{TakenDate:yyyy-MM-dd}\{OriginalName}
```

生成例:

```text
2026-07-12\DSC00001.ARW
```

## 2. 構文

### 2.1 基本形

```text
{Token}
{Token:Format}
{TakenDateInTimeZone:TimeZoneSpecifier|Format}
```

- トークンの正式表記は英語の PascalCase とする
- トークン名の入力は大文字・小文字を区別しない。保存時とUIの候補表示では正式表記へ正規化する
- `Format` は対応するトークンでのみ指定できる。`{SourceRelativeDirectory:N}` の `N` は末尾から残す階層数とする
- 同じトークンを複数回使用できる。ただし `{Sequence}` は最大1回とする
- トークンの前後に任意のリテラル文字を置ける
- 空のテンプレートは無効とする

例:

```text
{TakenDate:yyyy}\{TakenDate:MM}\{FileName}_{CameraModel}{Extension}
```

### 2.2 特殊文字のエスケープ

リテラルの波括弧は二重に記述する。

| 入力 | 出力 |
|---|---|
| `{{` | `{` |
| `}}` | `}` |

単独の `{` または `}`、閉じられていないトークンは構文エラーとする。

### 2.3 フォルダ区切り

- テンプレート内のフォルダ区切りは `\` のみとする
- `/` はWindowsのファイル名に使用できないため、区切りとして自動変換せず入力エラーとする
- 先頭・末尾の `\`、連続する `\\` による空のパス要素は無効とする
- 生成結果はコピー先ルートからの相対パスでなければならない
- ドライブ指定、UNCパス、先頭区切り、`.`、`..` のパス要素は禁止する

## 3. v1トークン

### 3.1 ファイルシステム系

これらのトークンだけを含む場合、Exifを読み込まない。

| トークン | 値 | Format | 例 |
|---|---|---|---|
| `{OriginalName}` | 元ファイル名。拡張子を含む | 不可 | `DSC_0101.NEF` |
| `{FileName}` | 元ファイル名から最後の拡張子を除いた部分 | 不可 | `DSC_0101` |
| `{Extension}` | 最後の拡張子。先頭の `.` を含む。拡張子がなければ空文字 | 不可 | `.NEF` |
| `{SourceRelativeDirectory}` | コピー元ルートから元ファイル格納フォルダまでの相対ディレクトリパス。コピー元ルート自体とファイル名は含まない | 階層数 | `DCIM\100MSDCF` |
| `{ModifiedDate}` | 元ファイルの最終更新日時 | 日時書式 | `20260712_142530` |
| `{FileSize}` | 元ファイルのバイト数を10進整数で表した値 | 不可 | `48201931` |
| `{Protected}` | ファイルの読み取り専用属性。読み取り専用なら `Protected`、そうでなければ `Unprotected` | 不可 | `Protected` |
| `{Sequence}` | 宛先競合を避ける連番。競合がなければ空文字、競合時は指定桁数。既定3桁 | 桁数 | （空文字）、`_001`、`_0001` |

`{FileName}{Extension}` は元の `{OriginalName}` と一致する。`photo.backup.jpg` の場合、`{FileName}` は `photo.backup`、`{Extension}` は `.jpg` とする。

#### `{SourceRelativeDirectory}`

コピー元を再帰走査したときのディレクトリ構造を、必要に応じてコピー先にも引き継ぐためのトークンとする。

- コピー元ルートがSDカードのルート `E:\`、元ファイルが `E:\DCIM\100MSDCF\DSC00001.ARW` の場合、値は `DCIM\100MSDCF` とする
- コピー元ルート直下のファイルでは空文字とする
- 値は0個以上の完全なパス要素であり、先頭と末尾に `\` を含まない
- 複数階層を保持し、単一のファイル名要素へ平坦化しない
- Format を省略した場合は全階層を出力する。`{SourceRelativeDirectory:N}` と正の10進整数を指定した場合は末尾から最大 `N` 階層を出力し、元の階層数が `N` 以下なら全階層を出力する
- 階層数は先頭ゼロや符号を含まない1以上の10進整数とし、`0`、負数、数字以外、前後の空白を含む指定は無効とする
- ファイルシステム列挙時に取得できるため、Exif読み込みは不要とする
- 各パス要素を個別に最終パスの検証対象とする。値に絶対パス、`.`、`..`、空の要素、コピー元ルート外を指す要素を許可しない

推奨例:

```text
{SourceRelativeDirectory}\{OriginalName}
```

```text
DCIM\100MSDCF\DSC00001.ARW
```

このトークンが空文字で、直後のリテラルが1個の `\` である場合、その `\` も出力しない。したがって同じテンプレートをコピー元ルート直下のファイルへ適用しても、`DSC00001.ARW` となり、先頭に空のパス要素は生じない。この省略規則は `{SourceRelativeDirectory}` の直後の区切り1個にだけ適用し、それ以外のトークンやリテラルには適用しない。

日付フォルダの下でコピー元の階層を保持する例:

```text
{TakenDate:yyyy-MM-dd}\{SourceRelativeDirectory}\{OriginalName}
```

異なるコピー元ルートを選択すれば値も変わる。たとえばカードのルートを選択した場合は `DCIM\100MSDCF`、`DCIM` 自体を選択した場合は `100MSDCF` となる。値の基準は常にスキャン時に選択したコピー元ルートとする。

たとえば相対ディレクトリが `DCIM\100MSDCF` の場合、`{SourceRelativeDirectory:1}` は `100MSDCF`、`{SourceRelativeDirectory:2}` 以上は `DCIM\100MSDCF` を出力する。階層選択の前に元の相対パス全体を検証し、不正な親参照などを階層選択によって隠すことはできない。

#### `{Protected}`

カメラの「プロテクト(保護)」操作は、DCF 規格によりファイルの読み取り専用属性として記録される。このトークンはスキャンのたびにファイル属性から直接取得する。

- Exif タグではないため、Exif 読み込みは不要とする
- Exif キャッシュには保存しない。属性の変更はファイルの更新日時を変えないため、キャッシュするとキーが一致したまま古い値を返す恐れがある
- ブール値であり「値がない場合」は存在しない。空文字ではなく必ず `Protected` / `Unprotected` のいずれかを出力する(空文字にすると単独使用時に空のパス要素が生じるため)
- Format は指定できない

`{Sequence}` はファイルシステムのメタデータだけでは確定せず、宛先の存在確認と同一スキャン内の割り当て結果を使って評価する。詳細は「4. 競合時の連番」を参照する。

### 3.2 Exif系

いずれかを含む場合、対象ファイルについて決定した「Exif 解析元」のメタデータ読み込みが必要となる。通常は対象ファイル自身を解析元とするが、設定「RAW+JPEGペアではJPEGのみ解析する」が ON の場合は、同じディレクトリ・同じベース名の有効な RAW+JPEG ペアについて JPEG を両方の解析元とする。ペア判定と曖昧候補の扱いは `DESIGN.md` の「RAW+JPEG ペアの Exif 解析」に従う。

| トークン | 値 | Format | 値がない場合 |
|---|---|---|---|
| `{TakenDate}` | Exifの撮影日時をタイムゾーン変換せず、記録された壁時計値のまま使用。サブ秒を含む(§5.1) | 日時書式 | Exif解析元のPCローカル更新日時へフォールバック |
| `{TakenDateLocal}` | Exifの撮影日時をPCの現在のタイムゾーンへ変換 | 日時書式 | Exif解析元のUTC更新日時をPCのタイムゾーンへ変換 |
| `{TakenDateInTimeZone}` | Exifの撮影日時を指定したタイムゾーンへ変換 | 3文字コードまたはUTC固定時差の `TimeZoneSpecifier` と任意の日時書式 | Exif解析元のUTC更新日時を指定タイムゾーンへ変換 |
| `{CameraMake}` | カメラメーカー名 | 不可 | `Unknown` |
| `{CameraModel}` | カメラモデル名 | 不可 | `Unknown` |
| `{CameraSerial}` | カメラボディのシリアル番号 | 不可 | `Unknown` |
| `{Lens}` | レンズモデル名 | 不可 | `Unknown` |
| `{Width}` | 向きを反映した画像の幅(ピクセル) | 数値書式 | `Unknown` |
| `{Height}` | 向きを反映した画像の高さ(ピクセル) | 数値書式 | `Unknown` |
| `{ExifWidth}` | Exifに記録された生の幅。向きを反映しない | 数値書式 | `Unknown` |
| `{ExifHeight}` | Exifに記録された生の高さ。向きを反映しない | 数値書式 | `Unknown` |
| `{Orientation}` | Exif Orientation の値(1〜8) | 数値書式 | `Unknown` |
| `{Aperture}` | 絞り値。既定は `F2.8` 形式 | 数値書式(指定時は `F` を付けない) | `Unknown` |
| `{ShutterSpeed}` | シャッタースピード。既定は `1-250s` 形式 | `1-250s` / `1-250` / `1_250s` / `1_250` | `Unknown` |
| `{ExposureTime}` | 露光時間の秒数(10進数値) | 数値書式 | `Unknown` |
| `{Iso}` | ISO感度(整数) | 数値書式 | `Unknown` |
| `{FocalLength}` | 焦点距離。既定は `35mm` / `23.5mm` 形式 | 数値書式(指定時は `mm` を付けない) | `Unknown` |
| `{FocalLength35mm}` | 35mm判換算焦点距離。既定は `35mm` 形式 | 数値書式(指定時は `mm` を付けない) | `Unknown` |
| `{Rating}` | 評価(スター)。`1`〜`5`、除外は `Rejected` | 数値書式(1〜5にのみ適用) | `Unknown` |
| `{HasGps}` | GPS位置情報の有無。`GPS` または `NoGPS` | 不可 | (常に値がある) |
| `{GpsLatitude}` | 緯度。既定は符号付き10進度・小数6桁 | `dms` / `dm` | `Unknown` |
| `{GpsLongitude}` | 経度。既定は符号付き10進度・小数6桁 | `dms` / `dm` | `Unknown` |
| `{GpsAltitude}` | 高度(メートル)。既定は `34.5m` 形式、海面下は負値 | 数値書式(指定時は `m` を付けない) | `Unknown` |

文字列メタデータは前後の空白を除去する。空文字または空白だけの値は「値がない」とみなす。数値系トークンも値がない場合は `Unknown` を出力する。

動画ファイル(MOV/MP4 等)は v1 では Exif 抽出の対象外とし、抽出項目をすべて値がない扱いにする。派生トークンである `{HasGps}` は欠落した緯度・経度から `NoGPS` を出力する。

#### 幅・高さ・向き

- フォーマットヘッダに記録された幅と高さ(JPEG の SOF、PNG の IHDR、HEIC の ispe 等)を `DecodedWidth` / `DecodedHeight` として抽出する。これらはフォーマットヘッダ由来の値だけとし、Exif の寸法タグで補完しない
- `{Width}` / `{Height}` の基礎寸法は、`DecodedWidth` と `DecodedHeight` の両方が有効ならその組を優先し、片方でも欠ける場合は `ExifWidth` と `ExifHeight` の両方が有効な組へフォールバックする。異なる取得元の幅と高さを混在させない。いずれも MetadataExtractor の通常解析で取得済みの情報であり、追加のデータ読み取りは発生しない
- TIFF ベースの RAW では IFD0 の寸法が埋め込みプレビューを指す機種がある。信頼できる実寸を特定できない場合は値がない扱いとし、誤った寸法を出力しない。取得元の優先順位は実機写真で検証する
- `{Width}`/`{Height}` は選択した基礎寸法に対し、Orientation が 5〜8(90度系の回転・反転)の場合に幅と高さを入れ替える。Orientation が欠落している場合は 1(回転なし)とみなす
- `{ExifWidth}`/`{ExifHeight}` は Exif タグの生値を入れ替えなしで出力する

#### シャッタースピードの表現

- 値がちょうど `1/N` 秒の場合は、既約分数を使い `1-250s` のように表す
- それ以外(1秒以上や `1/N` で表せない値)は `30s`、`2.5s`、`0.3s` のように10進数値+`s` で表す(最大6桁小数、末尾のゼロは出力しない)
- `{ShutterSpeed}` の書式指定は実例(1/250秒の見え方)による4種とし、完全一致で解釈する: `1-250s`(既定)、`1-250`、`1_250s`、`1_250`。区切り文字の指定は分数表現にのみ影響し、`s` の有無はすべての表現に適用する
- `{ExposureTime}` は同じ値を10進の秒数として出力する。既定は最大6桁小数・末尾ゼロなし(`0.004`、`0.000125`、`0.333333`)とし、数値書式で上書きできる。露光時間は内部的に有理数のまま保持し、丸めは出力時にだけ行う

#### 評価(Rating)

- 取得の優先順は、Exif IFD0 の `Rating`(0x4746)→ XMP の `xmp:Rating` → Exif の `RatingPercent`(0x4749)とし、有効な値が得られるまで次の取得元へフォールバックする
- `RatingPercent` は 1〜24→1、25〜49→2、50〜74→3、75〜98→4、99以上→5 に換算する
- 値 `0`、未設定、各取得元で定義された範囲外の値は、その取得元に候補がないものとして次の取得元を確認する。すべての取得元で候補がなければ「未評価」として値がない扱いにする
- XMP の `-1`(除外)は `Rejected` を出力する。数値書式は 1〜5 にのみ適用し、`Rejected` は書式にかかわらず固定文字列とする

#### GPS

- 緯度・経度・高度は GPS IFD から取得する。緯度と経度が両方存在し、緯度が -90〜90、経度が -180〜180 の範囲内で、かつ両方が0ではない場合だけ有効なGPS位置情報とする
- 緯度・経度の片方が欠落している、範囲外、または両方が0の場合は、緯度・経度・高度をすべて値がない扱いにする。高度だけを単独で保持・出力しない
- `{HasGps}` は緯度と経度が揃って有効な場合に `GPS`、それ以外は `NoGPS` を出力する
- 座標の既定は符号付き10進度・小数6桁とする(例: `35.681236`、`-118.242643`)。南緯・西経は負値
- 書式 `dms` は度・分・秒(小数1桁)+半球記号(例: `35-40-52.4N`)、`dm` は度・10進分(小数3桁)+半球記号(例: `35-40.874N`)とする。`:` や `/` を含まないパス安全な表記とする
- `{GpsAltitude}` は `GPSAltitude` と `GPSAltitudeRef` から求めたメートル値とし、海面下は負値とする

#### カメラシリアル番号

- Exif の `BodySerialNumber`(0xA431)を優先する。無い場合、メーカーノートにシリアル番号を記録する機種(Sony 等)では、MetadataExtractor が公開する値を取得できる場合にそれを使用する。取得可否と値の妥当性は実機写真で検証する

#### RAW+JPEG ペアでの適用

RAW+JPEG ペアで JPEG を解析元とした場合も、`{OriginalName}`、`{FileName}`、`{Extension}`、`{SourceRelativeDirectory}`、`{ModifiedDate}`、`{FileSize}`、`{Protected}` は評価対象ファイル自身の値を使用する。Exif 系トークンと、その撮影日時欠落時のフォールバック更新日時だけが解析元に従う。このため、JPEG と RAW の Exif 系評価は一致し得る一方、両者のファイル名、拡張子、サイズ、`{ModifiedDate}` は個別の値となる。

注意: JPEGのみ解析モードでは、寸法・向き・評価・GPS を含むすべての Exif 系トークンが解析元 JPEG の値になる。特に `{Width}`/`{Height}`/`{ExifWidth}`/`{ExifHeight}` は RAW 本来の寸法ではなく JPEG の寸法を返す。RAW 自身の値が必要な場合は「両方解析」モードを使用する。

### 3.3 数値書式

Format 列に「数値書式」とあるトークンには、.NET の標準・カスタム数値書式指定文字列(`D5`、`000`、`0.0` など)を指定できる。

- `CultureInfo.InvariantCulture` で評価する
- 書式を評価できない場合、または評価結果にパスで使用できない文字が含まれる場合は `InvalidNumberFormat` エラーとする
- 単位付き既定表記を持つトークン(`{Aperture}`、`{FocalLength}`、`{FocalLength35mm}`、`{GpsAltitude}`)は、数値書式を指定した場合に接頭辞・単位を付けず数値だけを書式で出力する(例: `{Aperture:0.0}` → `2.8`、`{FocalLength:000}` → `035`)
- 値がない場合は書式にかかわらず `Unknown` を出力する

## 4. 競合時の連番

### 4.1 出力形式

- 競合がなければ `{Sequence}` は空文字を出力する
- Formatには桁数を1〜9の10進整数で指定する。省略時は3桁とする
- `{Sequence}` は `_001`、`_002`、…、`_999` の順に候補を作る
- `{Sequence:4}` は `_0001`、`_0002`、…、`_9999` の順に候補を作る
- `_` はトークンの出力に含まれるため、テンプレート側に記述しない
- 指定桁数で表現できる最大値までを候補とする。たとえば2桁なら `_01`〜`_99`
- `0`、負数、10以上、数字以外、前後に空白がある指定は無効とする
- `{Sequence}` はテンプレート内に最大1回だけ記述できる
- 指定桁数の候補を使い切った場合は、自動解決せず競合エラーとする

推奨配置:

```text
{FileName}{Sequence}{Extension}
```

`{Sequence}` が空でも有効なパスになる位置へ配置しなければならない。たとえば `{Sequence}\{OriginalName}` は、競合がないと先頭のパス要素が空になるため無効とする。

### 4.2 割り当て手順

スキャン対象をコピー元ルートからの相対パスで、大文字・小文字を区別しない昇順に並べてから処理する。これにより、同じ入力集合では再スキャン時にも同じ割り当て順となる。

各コピー元ファイルについて次の順で候補を調べる。

1. `{Sequence}` を空文字として基本候補を生成する
2. 候補が同一スキャン内で別ファイルに予約されていれば、次の連番候補へ進む
3. 候補が実際の宛先に存在し、コピー元とコピー先のファイルサイズが一致し、かつコピー先ファイルシステムの精度で更新日時が一致すれば、その候補を「取込済」として予約する
4. 候補が存在しなければ、その候補を「未取込」として予約する
5. 候補が存在してサイズまたは更新日時が一致せず、テンプレートに `{Sequence}` がある場合は、上書き設定にかかわらず指定桁数の最初の連番候補から順に次の候補へ進む
6. 候補が存在してサイズまたは更新日時が一致せず、テンプレートに `{Sequence}` がない場合は、上書き設定が有効なら基本候補を「上書き対象」、無効なら「競合」とする
7. 空き、取込済、上書き対象、または競合が確定するまで繰り返す

更新日時の一致は、NTFS/ReFS では UTC の完全一致、exFAT では UTC の 10ms 単位、FAT/FAT32 では PC のローカル壁時計値の 2 秒単位で判定する。これ以外のコピー先ファイルシステムは非対応とし、スキャンを開始しない。サイズと更新日時が一致する別内容ファイルは識別できないため、内容同一性の完全な保証にはハッシュまたは取込履歴が別途必要となる。

同一スキャン内の予約済み候補は、ファイルサイズや更新日時が同じでも別ファイルへ再利用しない。これにより、異なる写真が1つの宛先へまとめて割り当てられることを防ぐ。

ファイルサイズと更新日時はプレビュー時のスナップショットとして保持する。コピー実行時は表示済みの割り当てを再計算せず、スナップショットと現在値を比較して変更の有無を検証する。コピー元またはコピー先が変化していた場合は、別の宛先へ自動変更せず、そのファイルを再スキャンが必要なエラーとする。

### 4.3 `{Sequence}` がない場合

テンプレートに `{Sequence}` がない場合は自動で連番を追加しない。基本候補にコピー元より古い既存ファイルがある場合、上書き設定が有効なら基本候補を上書き対象とし、無効なら競合とする。同一スキャン内で候補が重複した場合も競合とする。

上書き設定が割り当てへ影響するのは、テンプレートに `{Sequence}` がない場合だけとする。`{Sequence}` がある場合は、コピーが必要な既存候補を上書きせず、設定にかかわらず別の連番候補を割り当てる。

## 5. 日時・タイムゾーン

### 5.1 日時書式

- .NETのカスタム日時書式指定文字列を使用する
- 結果がPCの表示言語や地域設定で変わらないよう、`CultureInfo.InvariantCulture` で評価する
- 書式省略時の既定値は `yyyyMMdd_HHmmss` とする
- 撮影日時は Exif の `DateTimeOriginal` に `SubSecTimeOriginal`(サブ秒)を合成した値とし、`fff` などの書式でミリ秒以下を出力できる。サブ秒は7桁(100ナノ秒)まで保持し、超える桁は切り捨てる。サブ秒がない場合は0とする。合成後の値は `{TakenDateLocal}` と `{TakenDateInTimeZone}` の変換にも反映する
- 日時以外のトークンに `Format` を指定した場合はエラーとする
- 書式の評価結果にパスで使用できない文字が含まれる場合はエラーとする

代表例:

| 書式 | 出力例 |
|---|---|
| `{TakenDate:yyyy-MM-dd}` | `2026-07-12` |
| `{TakenDateLocal:yyyy-MM-dd_HHmmss}` | `2026-07-12_142530` |
| `{TakenDate:yyyyMMdd_HHmmss}` | `20260712_142530` |
| `{ModifiedDate:yyyy}\{ModifiedDate:MM}` | `2026\07` |

パイプを含む指定タイムゾーンの例:

```text
{TakenDateInTimeZone:JST|yyyy-MM-dd_HHmmss}
→ 2026-07-12_142530

{TakenDateInTimeZone:UTC+9|yyyy-MM-dd_HHmmss}
→ 2026-07-12_142530
```

`:` や `/` を含む書式は、Windowsパスとして不正な結果になるため使用しない。UIのテンプレート入力補助では安全な代表書式を提示する。

### 5.2 タイムゾーンの3方式

#### `{TakenDate}`: タイムゾーン未考慮

Exifの `DateTimeOriginal` に記録された年月日時分秒を、そのまま壁時計値として使用する。Exifにオフセットが記録されていても変換しない。従来の撮影日時別フォルダと同じ結果を得たい場合に使用する。

#### `{TakenDateLocal}`: PCのタイムゾーン

Exifの撮影日時とオフセットから瞬間を特定し、評価時点のPCのローカルタイムゾーンへ `TimeZoneInfo` で変換する。夏時間を含む変換規則はOSのタイムゾーンデータに従う。

#### `{TakenDateInTimeZone}`: 指定したタイムゾーン

構文:

```text
{TakenDateInTimeZone:TimeZoneSpecifier}
{TakenDateInTimeZone:TimeZoneSpecifier|Format}
```

- `TimeZoneSpecifier` には、アプリが定義する3文字のタイムゾーンコード、またはUTCからの固定時差を指定する
- 3文字コードと `UTC` 接頭辞は大文字・小文字を区別しない。保存時とUIの候補表示では大文字へ正規化する
- タイムゾーン指定子と日時書式は `|` で区切る
- 日時書式を省略した場合は `yyyyMMdd_HHmmss` を使用する
- `|` を記述した場合、その後の日時書式を空にはできない
- 未定義の3文字コードはエラーとする
- WindowsタイムゾーンIDやIANA IDをテンプレートへ直接指定する形式はv1では対象外とする

v1で定義するコード:

| コード | WindowsタイムゾーンID | 意味 |
|---|---|---|
| `UTC` | `UTC` | 協定世界時 |
| `JST` | `Tokyo Standard Time` | 日本時間 |
| `PST` | `Pacific Standard Time` | 北米太平洋時間 |
| `MST` | `Mountain Standard Time` | 北米山岳部時間 |
| `CST` | `Central Standard Time` | 北米中部時間 |
| `EST` | `Eastern Standard Time` | 北米東部時間 |
| `GMT` | `GMT Standard Time` | 英国時間 |
| `CET` | `W. Europe Standard Time` | 中央ヨーロッパ時間 |

3文字略称には地域によって意味が異なるものがあるため、アプリでは上表の固定マッピングを使用する。たとえば `CST` は中国標準時ではなく北米中部時間を表す。各Windowsタイムゾーンの夏時間規則はOSの `TimeZoneInfo` に従うため、`PST` を指定した場合も夏期は適切な夏時間オフセットで変換される。

UTC固定時差の構文:

```text
UTC
UTC+H
UTC-H
UTC+HH:MM
UTC-HH:MM
```

- `UTC+9`、`UTC-5`、`UTC+05:30`、`UTC+12:45` などを受け付ける
- 時差の範囲は `UTC-14:00`〜`UTC+14:00` とする
- 時は1桁または2桁、分を指定する場合は2桁で `00`〜`59` とする
- `UTC+14:00` と `UTC-14:00` が上限であり、14時間を指定した場合の分は `00` のみ有効とする
- 空白、秒、小数による指定は受け付けない
- 固定時差には夏時間を適用しない
- 保存時は `UTC+09:00` のように符号、2桁の時、2桁の分を含む形式へ正規化する。ゼロ時差は `UTC` へ正規化する

3文字コードの `JST` と固定時差の `UTC+9` は、現在はいずれもUTC+9となる。ただし、3文字コードは対応するWindowsタイムゾーンの規則、固定時差は常に指定されたオフセットを使用するという違いがある。

### 5.3 Exifオフセットとフォールバック

タイムゾーン変換には、Exifの `DateTimeOriginal` と `OffsetTimeOriginal` の組み合わせを優先して使用する。

- `DateTimeOriginal` と有効な `OffsetTimeOriginal` がある場合、`DateTimeOffset` として解釈して対象タイムゾーンへ変換する
- 撮影日時はあるがオフセットがない場合、瞬間を一意に決められない。この場合は撮影日時を対象タイムゾーンの壁時計値として扱い、時刻自体はずらさず `TakenDateOffsetMissing` 警告を返す
- 撮影日時自体がない場合、Exif解析元のUTC更新日時を対象タイムゾーンへ変換する。`{TakenDate}` ではExif解析元のPCローカル更新日時を使用する。通常は評価対象ファイルと解析元が同じだが、RAW+JPEG の JPEG のみ解析モードでは RAW の評価にも JPEG の更新日時を使う
- 不正なオフセット値は、オフセットなしと同じ扱いにして警告を返す

Exif キャッシュには、タイムゾーン変換前の壁時計値、オフセットの値と状態(有効・欠落・不正)など、上記の評価と警告を再現するための抽出結果を物理ファイル単位で保存する。更新日時へのフォールバック有無と警告は、撮影日時の欠落と Exif 解析元の更新日時から評価時に導出し、キャッシュには保存しない。変換後の日時、生成済みパス、警告文言、RAW と JPEG の対応関係も保存しない。キャッシュヒット時も同じ解析元決定と評価処理を通し、キャッシュミス時と生成パス・警告コードが一致しなければならない。JPEG のみ解析モードではペア RAW 自身のキャッシュを評価に使わず、JPEG キャッシュまたは JPEG の直接解析結果を使う。これによりキャッシュの有無や削除で生成パスが変わることを防ぐ。

PCのタイムゾーンやOSのタイムゾーン規則が変わると、`{TakenDateLocal}` の生成結果も変わり得る。コピー先を長期的に固定したい場合は、`{TakenDate}` または明示的な `{TakenDateInTimeZone}` を推奨する。

## 6. パス値の正規化と検証

### 6.1 リテラル部分

テンプレートに直接入力されたリテラルにWindowsパスの不正文字がある場合は、入力エラーとして利用者に修正を求める。暗黙の置換はしない。

禁止文字:

```text
< > : " / | ? *
```

制御文字(U+0000〜U+001F)も禁止する。

### 6.2 トークンから得た文字列

Exifなど利用者が直接修正できない値は、次の規則で安全なパス要素へ変換する。

1. 不正文字と制御文字を `_` に置換する
2. 連続して生成された `_` は1文字へまとめる
3. パス要素末尾の空白と `.` を除去する
4. 結果が空なら `Unknown` とする
5. `CON`、`PRN`、`AUX`、`NUL`、`COM1`〜`COM9`、`LPT1`〜`LPT9` と一致する場合は先頭に `_` を付ける。拡張子付きでも予約名判定を行う

元ファイル由来の `{OriginalName}`、`{FileName}`、`{Extension}` は通常Windows上で既に有効だが、同じ検証を最後に適用する。

`{SourceRelativeDirectory}` は複数のパス要素を生成できる唯一のv1トークンとする。他の文字列トークンと異なり、値全体を1個のファイル名としてサニタイズせず、コピー元ルート配下であることを確認したうえで各要素へ同じ検証を適用する。区切り文字そのものを `_` へ置換してはならない。

### 6.3 最終パス

展開後に以下を検証する。

- パス要素が空でない
- 各パス要素が `.` または `..` ではない
- コピー先ルートと結合し、正規化した絶対パスがコピー先ルート配下にある
- Windowsの予約デバイス名を含まない
- 宛先のフルパス長がアプリで定める上限以内である

パス長上限の具体値はコピー処理設計で決定する。テンプレートエンジンは、許容上限を呼び出し側から受け取って検証できる構造にする。

## 7. エラー

パーサーは例外メッセージだけで結果を返さず、UIがローカライズできる構造化エラーを返す。

最低限、次のエラーコードを定義する。

| エラーコード | 条件 |
|---|---|
| `TemplateEmpty` | テンプレートが空 |
| `UnclosedToken` | `}` がない |
| `UnexpectedClosingBrace` | 単独の `}` がある |
| `TokenNameEmpty` | `{}` または `{:...}` |
| `UnknownToken` | 未定義のトークン |
| `FormatNotSupported` | 非対応トークンに書式を指定 |
| `DuplicateSequenceToken` | `{Sequence}` が複数ある |
| `InvalidSequenceWidth` | `{Sequence}` の桁数が1〜9の整数ではない |
| `InvalidDateFormat` | 日時書式を評価できない |
| `InvalidNumberFormat` | 数値書式を評価できない、または結果に不正文字がある |
| `InvalidShutterSpeedFormat` | `{ShutterSpeed}` の書式が定義済みの4種のいずれでもない |
| `InvalidGpsFormat` | GPS座標トークンの書式が `dms` / `dm` のいずれでもない |
| `TimeZoneArgumentMissing` | `{TakenDateInTimeZone}` のタイムゾーン指定子がない |
| `InvalidTimeZoneCode` | タイムゾーンコードが3文字でない、または未定義 |
| `InvalidUtcOffset` | UTC固定時差の構文が不正、または許容範囲外 |
| `InvalidLiteralCharacter` | リテラルに不正文字がある |
| `InvalidPathStructure` | 絶対パス、空要素、`.`、`..` など |
| `ReservedDeviceName` | 生成後も予約名が残る |
| `PathTooLong` | 許容パス長を超える |
| `SequenceExhausted` | 指定桁数で利用可能な連番候補がない |

エラーにはコードに加え、テンプレート内の開始位置と長さ、必要であればトークン名を含める。日本語・英語の文言はUI側の `.resx` から生成する。

警告コードとして最低限 `TakenDateOffsetMissing` と `TakenDateFallbackToModifiedDate` を定義する。警告はプレビューに表示するが、スキャンとコピーは続行できる。

## 8. 評価結果

テンプレート評価は少なくとも次を返す。

- 生成された相対パス
- 生成された絶対宛先パス
- Exif読み込みが必要か
- 確定した連番文字列。競合がなければ空文字
- 競合回避によって宛先名が変更されたか
- 撮影日時トークンが更新日時へフォールバックしたか
- タイムゾーン変換に使用した3文字コードとWindowsタイムゾーンID、または正規化済みUTC固定時差、およびExifオフセット
- メタデータ値のサニタイズが発生したか
- 警告とエラーの一覧

テンプレート解析結果は再利用可能な中間表現として保持する。プレビュー更新のたびに同じ文字列を再解析せず、テンプレートが変更された場合だけ再解析する。

テンプレート評価は、ファイル情報だけで基本相対パスを作る処理と、宛先状態を使って `{Sequence}` を確定する処理の2段階に分ける。

## 9. UIと多言語対応

- 対応言語は `ja-JP` と `en-US`
- OSのUI言語に応じて初期言語を選ぶ
- 日本語・英語以外では `en-US` をフォールバックとする
- UI上で言語を変更できるようにするかは、設定画面の設計時に決定する
- トークン名、日時書式、保存済みテンプレートは翻訳しない
- トークン候補にはローカライズした説明と、英語の正式トークン名を併記する
- 入力中はデバウンスして解析し、エラー位置とローカライズした説明を表示する
- 無効なテンプレートではスキャンとコピーを開始できない

## 10. 初期単体テスト項目

### 正常系

1. `{OriginalName}` が拡張子を含む元ファイル名になる
2. `{FileName}{Extension}` が元ファイル名を再構成する
3. 複数ドットを含むファイル名を最後の拡張子で分割する
4. 拡張子がないファイルで `{Extension}` が空になる
5. 日時の既定書式と指定書式を `InvariantCulture` で評価する
6. `{{` と `}}` がリテラルの波括弧になる
7. トークン名を大文字・小文字を変えても同じ結果になる
8. Exifトークンの有無を解析時に判定できる
9. Exif撮影日時がない場合に更新日時へフォールバックする
10. Exif文字列の不正文字と予約名が安全に変換される
11. `{TakenDate}` がExifオフセットを無視して壁時計値を維持する
12. `{TakenDateLocal}` がExifオフセットからPCタイムゾーンへ変換する
13. `{TakenDateInTimeZone:JST|...}` が固定マッピングを使って変換する
14. `{TakenDateInTimeZone:UTC+9|...}` がUTC+9の固定時差で変換する
15. `UTC+05:30` と `UTC+12:45` の分単位の固定時差を変換する
16. UTC固定時差を正規化しても同じ変換結果になる
17. Exifオフセット欠落時に時刻をずらさず警告を返す
18. Exif撮影日時欠落時に更新日時を対象タイムゾーンへ変換する
19. 競合がない場合に `{Sequence}` が空になる
20. `{Sequence}` の既定値で `_001` が付く
21. `{Sequence:4}` で `_0001` が付く
22. `_001` が使用済みなら `_002` が付く
23. 連番候補の既存ファイルがコピー元と同じサイズかつファイルシステム精度で同じ更新日時なら、その番号で取込済になる
24. `{Sequence}` がある場合、サイズまたは更新日時が異なれば、上書き設定が有効でも次の連番候補が割り当てられる
25. `{Sequence}` がある場合、サイズまたは更新日時が異なれば、上書き設定が無効でも次の連番候補が割り当てられる
26. `{Sequence}` がなくサイズまたは更新日時が異なる場合、上書き設定が有効なら基本候補が上書き対象になる
27. `{Sequence}` がなくサイズまたは更新日時が異なる場合、上書き設定が無効なら競合になる
28. 同一バッチで同じ候補になるファイルへ異なる番号が割り当てられる
29. コピー元の列挙順が違っても、相対パスによる割り当て結果が同じになる
30. `{SourceRelativeDirectory}` が1階層および複数階層の相対ディレクトリを保持し、`{SourceRelativeDirectory:N}` が末尾の最大N階層を選択する
31. コピー元ルート直下のファイルでは `{SourceRelativeDirectory}` が空になり、直後の区切りも省略される
32. 異なるサブフォルダに同名・同サイズのファイルがあっても、`{SourceRelativeDirectory}` を含むテンプレートでは別の宛先へ割り当てられる
33. JPEGのみ解析モードで、ペアのJPEGから取得したExif値がJPEGとRAWの両方へ適用される
34. JPEGのみ解析モードでも、RAWのファイルシステム系トークンはRAW自身の値になる
35. JPEGに撮影日時がない場合、ペアRAWのExif系撮影日時もJPEGの更新日時へフォールバックする
36. 両方解析モードではJPEGとRAWがそれぞれ自身のExifとフォールバック更新日時を使用する
37. 単独RAWはJPEGのみ解析モードでもRAW自身を解析元とする
38. JPEGのみ解析モードでRAW自身のキャッシュが存在しても使用せず、JPEGキャッシュの有無で生成パスと警告コードが変化しない
39. 大文字・小文字の違い、別ディレクトリ、複数候補、未対応拡張子について仕様どおりにペア判定する
40. Exif解析を並列化しても相対パス順による連番割り当てが変化しない
41. `{Width}`/`{Height}` が Orientation 5〜8 で幅と高さを入れ替え、`{ExifWidth}`/`{ExifHeight}` は生値を維持する
42. `DecodedWidth` / `DecodedHeight` がフォーマットヘッダ由来の値だけを保持し、両方が有効ならその組、片方でも欠ければ Exif の幅・高さの組へフォールバックし、異なる取得元を混在させない
43. `SubSecTimeOriginal` が `{TakenDate:HHmmss.fff}` などに反映され、7桁を超えるサブ秒は切り捨てる
44. `{ShutterSpeed}` が `1/N` 秒を `1-250s`、それ以外を `30s`・`2.5s`・`0.3s` 形式で出力し、4種の書式が機能する
45. `{ExposureTime}` の既定出力(最大6桁小数・末尾ゼロなし)と数値書式指定を評価する
46. `{Aperture}`・`{FocalLength}`・`{FocalLength35mm}`・`{GpsAltitude}` の単位付き既定表記と、数値書式指定時の単位なし出力を評価する
47. `{Iso}` などの数値トークンにゼロ埋め等の数値書式を適用できる
48. Rating が Exif → XMP → `RatingPercent` の優先順で有効値までフォールバックし、`0`・未設定・範囲外を候補なし、XMP の `-1` を `Rejected` として扱う
49. GPS座標の既定書式・`dms`・`dm` を評価し、緯度・経度の組が欠落・範囲外・ともに0の場合は高度も含めて GPS なし扱いになる
50. `{HasGps}` が `GPS` / `NoGPS` を出力する
51. `{GpsAltitude}` が `GPSAltitudeRef` に従って海面下で負値になる
52. `{CameraSerial}` が `BodySerialNumber` を優先し、無ければメーカーノート由来の値を使う
53. `{Protected}` が読み取り専用属性から `Protected` / `Unprotected` を出力し、Exif 読み込みを発生させない
54. JPEGのみ解析モードで寸法・向き・評価・GPS も JPEG の値が RAW へ適用され、`{Protected}` は各ファイル自身の属性を使う

### 異常系

1. 空テンプレート
2. 未知のトークン
3. 閉じていないトークンと単独の閉じ波括弧
4. 非日時トークンへの書式指定
5. 先頭・末尾・連続するパス区切り
6. `/`、絶対パス、ドライブ指定、UNCパス
7. `.` と `..` のパス要素
8. リテラルに含まれる不正文字
9. 許容上限を超える宛先パス
10. 正規化後のパスがコピー先ルート外へ出ないこと
11. `{Sequence}` の複数指定と無効な桁数指定
12. `{Sequence}` が空のときに空のパス要素が生じる配置
13. 指定桁数の連番を使い切った場合
14. `{TakenDateInTimeZone}` のタイムゾーン指定子欠落、3文字以外、未定義コード、空の日時書式
15. UTC固定時差の範囲外、不正な分、空白、秒、小数による指定
16. `{SourceRelativeDirectory}` の階層数が0、負数、先頭ゼロ付き、数字以外、前後に空白を含む場合
17. コピー元ルート外、絶対パス、`.`、`..`、空の要素を含む相対ディレクトリ値
18. 数値書式を評価できない場合と、評価結果に不正文字が含まれる場合
19. `{ShutterSpeed}` と GPS座標トークンへの未定義の書式指定
20. 数値書式非対応トークン(`{CameraSerial}`、`{Protected}` 等)への書式指定

## 11. 実装境界

ステップ1では以下を実装する。

- テンプレートの字句解析と構文解析
- ファイルシステム系トークンの評価
- コピー元ルートからの相対ディレクトリ計算と `{SourceRelativeDirectory}` の評価
- 日時書式
- 相対パスの生成、正規化、検証
- `{Sequence}` の候補生成と、抽象化した宛先照会を使う競合解決
- 構造化エラー
- 上記に対する単体テスト

Exif系トークンはパーサーが認識し、Exifが必要であることを判定できるところまで含める。実際のMetadataExtractor連携とExif値の取得は開発ステップ5で実装する。
